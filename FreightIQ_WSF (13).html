Strategic context

The problem

  • Manual FMCSA, lane history, and rate checks don’t scale when lanes move weekly.
  • Teams re-key the same data across spreadsheets, slowing bids and increasing error.
  • Cold-chain needs both cost and service risk in one pass.

Agentic design (FreightIQ)

  • Deterministic engine scores Cost, Balanced, and Reliability in parallel on the same eligible pool.
  • Reasoning agent (Llama 3.3 70B via Groq) reads the structured matrix and writes a grounded recommendation — real LLM call, streamed live.
  • Backend deployed on Render (Node.js + Express); model layer is swappable to Claude or GPT with a one-line endpoint change.
  • Agent is constrained to cite specific carriers/numbers from the matrix; cannot hallucinate carriers or rates.
  • No re-query on persona switch; agent runs once per search.
  • Intended to feed a workflow / TMS handoff after human review of Top 3.

FAQ

Quick reference for how FreightIQ scores carriers, applies gates, and simulates tendering.

What is deadhead?

Deadhead is the empty miles a carrier runs to reach your WSF complex (pickup) before the loaded leg. FreightIQ penalizes long repositioning moves because they add cost, time, and failure risk before the shipment ever moves.

How are personas calculated?

All personas score the same eligible pool with different weights. Money Saved (Cost optimized) pushes most of the signal into effective $/mi and light safety/on-time touches. Peace of Mind (Reliability tier) shifts heavily toward safety and fleet on-time %, with rate as a smaller slice. Balanced spreads weight across lane proof, safety, on-time, rate, claims, and deadhead—including roughly 15% deadhead after normalization—so empty-mile risk shows up in the default view.

What does the Weather Alert do?

Simulate Weather Alert applies the same high-reliability gate used in the Reliability column to all three persona views: carriers must meet safety ≥ 85 and fleet on-time ≥ 90% before they appear in any column. That keeps Cost, Balanced, and Reliability comparable under a hardened pool when you want to stress service.

How do I tender a load?

Click Tender load on any ranked carrier card. FreightIQ runs a mock EDI 204 (Motor Carrier Shipment Information) sequence: draft tender, carrier acknowledge, and completion toast. The Route tab then locks the map and animation to the awarded carrier (any rank you chose), not only the #1 row.

What makes FreightIQ "agentic" rather than just a scoring tool?

FreightIQ pairs a deterministic engine (transparent weighted math over the carrier pool) with an LLM reasoning agent that reads the structured scoring matrix and produces a grounded recommendation in real time. The engine handles the math because rate calculations and gates need to be auditable; the agent handles the reasoning — what trade-off to surface, which persona to recommend, what risk to flag — because that's where language models add real value. This split is how production agentic systems are actually architected: agents call deterministic tools, they don't replace them.

Can the reasoning agent hallucinate carriers or rates?

No. The agent only sees the JSON scoring matrix produced by the deterministic engine — actual carriers, actual rates, actual safety scores. The system prompt requires it to cite specific carriers and numbers from the matrix, and it has no other source of carrier data to draw from. If a carrier or rate isn't in the matrix, the agent literally cannot mention it. This is grounded generation, not free-form recall.

Why Wayne Sanderson Farms specifically?

WSF moves frozen poultry on temperature-controlled freight, which puts the use case at the intersection of three high-stakes constraints: USDA cold-chain compliance (a temperature deviation can trigger a recall), retail DC arrival windows (late equals chargeback), and reefer reliability (a breakdown is operationally and financially catastrophic). That makes the case for paying a premium for reliability defensible in a way it isn't on dry van — the agent's reasoning has real domain stakes to weigh.

Why are there three personas instead of one optimal answer?

Real procurement decisions involve trade-offs leadership needs to see, not a single black-box recommendation. Cost Optimized shows the cheapest defensible carrier; Reliability Tier shows the safest pick gated to safety ≥ 85 and on-time ≥ 90; Balanced Fit shows the dollar-per-risk sweet spot. Surfacing all three on the same eligible pool — without re-querying — gives a logistics director the trade space at a glance and lets the agent's recommendation carry context about why Balanced (or Reliability) wins for cold-chain, not just which.

How is the API key kept secure?

The LLM API key never touches the browser. Requests go from the FreightIQ frontend to a Node.js + Express backend deployed on Render, which holds the key as an environment variable on the server and forwards authenticated requests to the model provider. The frontend code is fully open-source on GitHub; the key is not. This is the standard production pattern for any LLM-powered web app.

Why Groq + Llama 3.3 70B instead of GPT-4 or Claude?

Two reasons. First, cost: Groq's free tier handles the prototype's volume without burning credits, which keeps the project deployable as a student project. Second, latency: Groq's hardware delivers sub-second response times, which matters when a broker is watching reasoning stream in during a live demo. The model layer is intentionally swappable — a single endpoint change in the backend would point at Claude or GPT-4 if the use case demanded it. The reasoning quality on a structured task like this is competitive across all three.

What happens if the reasoning agent fails or times out?

FreightIQ has a graceful fallback: if the live LLM call errors, times out, or hits a rate limit, the reasoning log silently switches to a deterministic trace (citing the same matrix data) and the LIVE badge turns to FALLBACK. The deterministic results — Top 3 / Bottom 3 per persona, route, opportunity costs — are unaffected because they don't depend on the LLM at all. The agent is an enhancement, not a single point of failure.

Why does the deadhead penalty matter?

Deadhead is empty miles a carrier drives to reach the pickup. On paper a carrier might bid the lowest rate, but if their nearest available truck is 300 miles from the WSF complex, the broker is paying for an empty repositioning leg in service quality and timing risk. Cold chain amplifies this: a tired driver who burned a full shift deadheading is more likely to miss a DC window. Balanced Fit includes roughly 15% deadhead weight after normalization specifically to surface this hidden cost in the default view.

What data is real vs. synthetic in this prototype?

Real external integrations: the live LLM call to Groq for the reasoning agent, and OpenStreetMap tiles for the Route view. Synthetic: the carrier dataset, per-mile rates, lane history, and weather state — all generated client-side to be representative of southeast cold-chain freight. Replacing synthetic with real is a data-source swap, not an architecture change: the engine consumes carrier objects with FMCSA-style fields, so plugging in FMCSA SAFER, a TMS feed, or a load-board API is well-understood integration work.

Could this run on a real TMS feed instead of synthetic carriers?

Yes — the architecture was designed for it. The deterministic engine consumes a list of carrier objects with FMCSA-style fields (DOT number, safety score, fleet on-time, claims rate, equipment, home base). Replacing the synthetic dataset with a live feed from a TMS, FMCSA SAFER, DAT RateView, or a load board API is a data-source swap, not an architecture change. The agent layer doesn't care where the matrix comes from — only that it's structured, accurate, and current.

Where is the source code?

Public on GitHub at github.com/BIGMAN7474/Freight-IQ-Agent. The repo includes the Express backend (index.js), this single-file frontend, package.json for Node dependencies, and a README with the architecture diagram. The backend deploys automatically to Render on every push to main; the frontend deploys automatically to GitHub Pages.

    Prototype build — full disclosure

    Carrier pool. 100 carriers from the GenLogs Top 100 Carriers Q1 2025 list (sensor-network observation ranking) plus 3 additional cold-chain capable carriers (Heartland Express, Dart Transit, Decker Truck Line) added for reefer coverage — 103 total, all verified against FMCSA SAFER.

    What's real federal data. On page load, the app hydrates each carrier with real federal data from FMCSA's QCMobile API — safety rating, operating status, insurance, and CSA BASIC measures — and marks each verified carrier with a green ✓ FMCSA badge. Weather risk per WSF lane state is observed autonomously every 15 minutes by a pg_cron observer in Supabase calling NOAA's federal active-alerts API.

    Carrier gate. The pool is for-hire asset-based carriers only — no private fleets (Walmart, UPS, FedEx, Tyson, US Foods), no 3PL / asset-light brokers (Hub Group, Penske Logistics, NFI, Landstar, DB Schenker, Ryder Integrated, Cardinal). Those were removed from the pool entirely so every carrier shown is one a Wayne-Sanderson dispatcher could actually tender. The gate still enforces FMCSA safety floors (no Conditional or Unsatisfactory rating), insurance currency, equipment-type match, and (for Reliability persona or weather mode) elevated safety/on-time minimums.

    Honest caveats. Severity-to-event mapping in the NOAA observer is a heuristic operational-risk assignment and would be calibrated against historical disruption data in production. The reasoning panel (Llama 3.3 70B via Groq) generates narrative explanations of carrier rankings but does not influence the deterministic scoring engine. Claims history and reefer PM remain synthetic at this prototype stage — those require a paid CSA SMS feed and internal claims data to operationalize.
